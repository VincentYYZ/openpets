你要做的，其实分两层
如果你想“通过不同的输入来设计动作”，建议先把问题拆成两层：

输入层
什么事件会触发动作
比如：拖拽、命令执行、权限等待、任务成功、报错、手动点击、定时器、窗口移动
动作层
这些输入最终想让宠物表现成什么
比如：waving、waiting、jumping、failed、run-left、run-right
在这个项目里，不要直接把“输入”硬编码成“动画”，而是先归一化成“语义状态”，再交给动画系统。

当前项目的动作链路
你现在这套桌面宠物，基本是这条链：

输入事件
来自主进程、IPC、拖拽、计时器、agent 状态
行为/反应状态
PetBehaviorMode
OpenPetsReaction
动画决策
apps/desktop/src/pet-animation-resolver.ts
动画映射
apps/desktop/src/reaction-animation-mapping.ts
最终渲染
apps/desktop/src/pet-window.ts
用 spritesheet + CSS steps() 播放
所以如果你要扩展，核心工作是改这几层，不只是改一处。

推荐你的设计方式
方案一：先复用已有动作
这是成本最低的做法。

比如你新增一种输入：

用户拖入文件夹
agent 开始执行命令
agent 等待用户授权
agent 执行成功
agent 执行失败
你不一定要新增动画类型，可以先把它们映射到现有动作：

开始执行 -> running
等待确认 -> waiting
成功 -> jumping
失败 -> failed
提醒/关注 -> waving
你主要需要改：

输入接入点
状态归一化
reaction 映射
对应文件：

apps/desktop/src/default-pet-controller.ts
apps/desktop/src/pet-behavior-machine.ts
apps/desktop/src/local-ipc-protocol.ts
apps/desktop/src/reaction-animation-mapping.ts

这是成本更高的做法。

比如你想新增：

sleeping

你需要同时做这些事：

UniversalSpriteState

"sleeping"

3. 更新 spritesheet 资源
这是最关键的。

当前默认宠物和已安装宠物，本质都依赖统一的 spritesheet 结构。
如果你加了一个新动作行，就意味着：

默认宠物的 spritesheet.webp 要补新行
已安装宠物如果也要支持，就也要补同样的新行
否则会出现“逻辑有这个动作，但资源画不出来”的问题
所以：

改映射很容易
改动作资源协议才是真正的大工程
你应该优先设计“输入到语义”的中间层
这是最重要的设计建议。

这样以后你要换动画风格，只改映射，不用改所有输入源。

在这个仓库里，具体要改哪些地方
1. 定义新的输入来源
如果输入是新的系统事件，你要先找到入口。

常见入口：

拖拽/窗口交互
apps/desktop/src/pet-window.ts
默认宠物行为
apps/desktop/src/default-pet-controller.ts
状态机
apps/desktop/src/pet-behavior-types.ts
apps/desktop/src/pet-behavior-machine.ts
Agent/IPC 事件
apps/desktop/src/local-ipc-protocol.ts
相关 controller / preload
2. 把输入归一化成状态
你需要判断这个输入属于哪一类：

移动类
影响 PetBehaviorMode
如 walk-left / walk-right
表情/反应类
影响 OpenPetsReaction
如 thinking / waiting / success / error
一般来说：

短时情绪/任务反馈 -> 放进 OpenPetsReaction
持续性移动行为 -> 放进 PetBehaviorMode
3. 调整优先级规则
在 pet-animation-resolver.ts 里，当前已经有优先级逻辑：

暂停时，优先反应
有显式 reaction 时，优先 reaction
否则按走路状态决定 run-left/run-right
如果你加了新输入，往往还要想清楚：

它是否压过移动动作
它持续多久
是否可打断
是否播完后回到 idle / walking
这部分通常改：

apps/desktop/src/pet-animation-resolver.ts
apps/desktop/src/default-pet-controller.ts
4. 做动画映射
如果只是复用已有动作：

改 defaultReactionToSpriteState
如果新增动作：

改 UniversalSpriteState
改 defaultPetSprite.states
改 spritesheet 资源
5. 做渲染验证
最终动画显示在：

apps/desktop/src/pet-window.ts
这里会根据：

data-reaction-state
data-motion-state
生成对应 CSS 规则。

所以最后一定要验证：

状态有没有发出去
CSS 有没有切换到正确行
帧数和 duration 对不对
动画优先级是否符合预期
我建议你按这个优先级做
第一阶段：只加“输入 -> 已有动作”
先不要新增新动画资源，只做：

新输入接入
归一化成 OpenPetsReaction
映射到已有动作
这样最快看到效果，风险最低。

第二阶段：确认哪些输入真的需要独立动作
只有在你确认这些动作明显不够表达时，再新增：

新 UniversalSpriteState
新 spritesheet 行
新资源协议
第三阶段：做可配置化
如果你希望未来更灵活，可以把：

输入 -> reaction
reaction -> animation
做成可配置，而不是全部写死在代码里。

这个项目其实已经有一点这个方向了，因为：

reactionAnimationOverrides 已经支持用户覆盖 reaction 对应的动画
一个很实用的设计模板
你可以按这个表来设计：

输入	语义状态	动画	持续策略
命令开始执行	working	running	直到命令结束
等待用户授权	waiting	waiting	直到授权完成
命令成功	success	jumping	播 1 次后恢复
命令失败	error	failed	播 1 次后恢复
用户点击宠物	attention	waving	短暂播放
拖入文件夹	attention / working	waving / running	视业务而定
如果你要实现“新输入驱动动作”，最低工作量版本
最小落地步骤是：

新增输入事件
在 controller 里把它转成 OpenPetsReaction
必要时设置一个 timeout
让 pet-animation-resolver.ts 接管优先级
在 reaction-animation-mapping.ts 里复用现有动画
这样你几乎不用碰资源文件。

总结
如果你想通过不同输入来设计动作，主要要做的是：

先定义输入来源
把输入归一化成语义状态
把语义状态映射到动作
处理优先级、持续时间、恢复逻辑
如果要新增真正的新动作，再改 spritesheet 协议和资源
最推荐的路线是：

先复用现有动作
先跑通输入 -> reaction -> animation
最后再决定要不要新增动作资源
